Motivation 0:04
Why generative models
Discriminative
Model the posterior p(y | x): the probability of a label y given an example x. Great for classification, useless for creating new x.
Generative
Model the likelihood p(x | y) and the marginal p(x): the probability of the data itself. This is what lets you sample new examples.
Why diffusion, specifically
  • State of the art for images, audio, and video: Stable Diffusion, Imagen, Sora
  • Trains with a simple, stable objective: predict the noise, plain MSE
  • No adversarial training, no mode collapse: just gradient descent
  • The math is approachable enough to implement end-to-end in an afternoon

Session 1 · NB01 0:09
JAX in a nutshell

A differentiable NumPy-like library that runs on accelerators. Python code is traced into an intermediate representation that enables domain-specific compilation. Designed to be functional, so it can transform your code, not just run it.

Common transformations
  • grad(): computes the gradient of a numerical function
  • jit(): compiles operations together via XLA
  • vmap(): automatic vectorisation / batching
  • pmap(): automatic parallelisation across devices

Next four slides: one transformation each, in depth.

NumPy
Mutable arrays · eager execution · CPU only · no built-in autodiff
JAX
Immutable arrays · traced & compiled (jit) · CPU/GPU/TPU · grad built in
import jax.numpy as jnp
# same API as np, different runtime
Session 1 · NB01 0:11
grad: automatic differentiation
  • Takes a numerical function, returns a new function computing its gradient
  • Analogous to the ∇ operator from vector calculus
  • By default differentiates w.r.t. the first argument: pass argnums to control which
  • Need the value and the gradient? Use jax.value_and_grad: avoids a second forward pass
In JAX
def loss_fn(params, x, y):
  pred = model(params, x)
  return jnp.mean((pred - y) ** 2)

grad_fn = jax.grad(loss_fn)
grads = grad_fn(params, x, y)
# grads has the same pytree shape as params
value_and_grad
loss, grads = jax.value_and_grad(loss_fn)(params, x, y)
Session 1 · NB01 0:13
jit: the compiler
  • Compiling JAX code makes it performant. The same code runs on CPU, GPU, or TPU unchanged
  • Transforms Python into an intermediate representation, jaxpr, via tracing
  • The default tracing level is ShapedArray: each tracer has a concrete shape but no concrete value
  • It does not capture side effects: print, list mutation, etc. only run during the trace, not on later calls
First call traces, then compiles
@jax.jit
def f(x):
  print("tracing!") # only once per shape
  return x ** 2

f(jnp.ones(3)) # traces, compiles, runs
f(jnp.ones(3)) # cached: fast
f(jnp.ones(5)) # new shape → re-traces
Session 1 · NB01 0:15
vmap & pmap: batching and scale
vmap: vectorisation
Transforms a function to operate over a batch dimension without writing a loop. in_axes/out_axes pick which axis is the batch, per argument.
f = jax.vmap(single_fn, in_axes=(0, None))
f(x_batch, shared_param)
pmap: parallelisation
Same idea, across devices instead of within one: replicates the function on every GPU/TPU and splits the batch across them.
f = jax.pmap(train_step)
f(params, x_sharded) # one shard per device

In this tutorial: vmap batches q_sample / predict_noise over images. pmap isn't needed at Colab scale, but the exact same pattern scales training to a full TPU pod.

Session 1 · NB01 0:17
PRNG: explicit randomness
Why not NumPy-style global state?
  • Global mutable random state is a side effect: incompatible with jit's purity requirement
  • JAX makes randomness explicit: every call takes a key and returns deterministic output for that key
  • Same key + same call ⇒ same numbers, always: reproducible by construction
The splitting discipline
key = jr.PRNGKey(0)
key, sub1, sub2 = jr.split(key, 3)
noise = jr.normal(sub1, (32, 32, 3))
mask = jr.bernoulli(sub2, 0.1, (8,))
  • Never reuse a key: split before every independent draw
  • Carry the fresh key forward, consume sub* once
  • jr.fold_in derives a key from an int (e.g. a step counter) without a full split

Session 1 · Forward 0:24
Diffusion models

A diffusion model defines a Markov chain of steps that slowly add random noise to data, then learns to reverse that process to construct new samples from noise.

Two processes
  • Forward process q: gradually adds noise to an image, on a fixed schedule, until it becomes pure noise
  • Backward process p: a learned denoising process that removes noise from an image, step by step
x0 ——— add noise ———► x1 → ... → xT (clean) (noise) xT ——— denoise ————► xT−1 → ... → x0 (noise) εθ(xt, t) (clean) Forward: q(xt | xt−1) = 𝒩(√αt xt−1, βt I) Reverse: pθ(xt−1 | xt) ← learned

Markov chain of forward (reverse) diffusion (Sohl-Dickstein et al. 2015; Ho et al. 2020)

Session 1 · Forward 0:26
The forward process
  • Sample a real image x0 ~ q(x)
  • Add a small amount of Gaussian noise, T times, producing a sequence x1, ..., xT
  • Step sizes are controlled by a variance schedule βt
  • As t grows, the sample loses its distinguishable features: in the limit, xT is just Gaussian noise
Nothing to learn here
The forward direction is a fixed formula, applied one step at a time: no network, no training. It's the reverse direction, next, that has to be learned.
Session 1 · Forward 0:28
The backward process
  • p(xt−1 | xt) is intractable: it requires knowing the distribution of all possible images
  • So we approximate it with a neural network pθ
  • Assuming the reverse step is Gaussian too, we'd need to estimate a mean and a variance, both parametrized by θ
The reparametrization trick
Instead of predicting the mean directly, reparametrize it so the network predicts the added noise εθ(xt, t) for a given noise level t. This is exactly the term that appears in the (simplified) training loss.
Session 1 · Forward 0:30
Training & inference
Algorithm 1: Training
repeat
  x0 ~ q(x0)
  t ~ Uniform({1, ..., T})
  ε ~ 𝒩(0, I)
  take gradient step on
    ∇θ ‖ ε − εθ(√ᾱtx0+√1−ᾱtε, t) ‖2
until converged
Algorithm 2: Sampling
xT ~ 𝒩(0, I)
for t = T, ..., 1 do
  z ~ 𝒩(0, I) if t > 1 else z = 0
  xt−1 = 1/√αt ( xt − 1−αt√1−ᾱt εθ(xt,t) ) + σtz
end for
return x0

Same shape you'll see in the notebooks: train_step is Algorithm 1, ddpm_sample is Algorithm 2: with the schedule swapped in for αt, ᾱt.

Session 1 · Forward 0:32
The closed-form jump

Applying the forward step 1000 times is slow. But because Gaussians compose, we can jump from x0 directly to any xt:

xt = √ᾱt · x0 + √1 − ᾱt · ε ε ~ 𝒩(0, I)     ᾱt = ∏s≤t αs   (cumulative product)
Why this matters for training
  • Sample any t uniformly during training
  • Compute xt in one operation: no loop
  • Training scales to T=1000 without T× slowdown
What ᾱt encodes
  • At t=0: ᾱt ≈ 1 → almost pure signal
  • At t=T: ᾱt ≈ 0 → almost pure noise
  • The schedule shapes how fast it decays
Session 1 · Forward 0:35
Noise schedules
Linear (Ho et al. 2020)
Specify βt directly. Linearly spaced from βstart to βend.
betas = linspace(βstart, βend, T)
alphas = 1 − betas
ᾱt = cumprod(alphas)
⚠ degrades at small T; β must scale with T.
Cosine (Nichol & Dhariwal 2021)
Specify ᾱt via cosine. Derive βt from consecutive ratios.
ᾱt = cos2((t/T + s)/(1+s) · π/2)
betas = 1 − ᾱt / ᾱt−1
betas = clip(betas, 0, 0.999)
✓ T-invariant. s=0.008 prevents β1 ≈ 0.

We use the cosine schedule throughout: it trains more stably and is T-invariant.

Session 1 · Reverse 1:00
The reverse problem
  • q(xt−1 | xt) is intractable: we don't know the data distribution
  • q(xt−1 | xt, x0) is Gaussian: conditioning on x0 makes it tractable
  • So: train a network to estimate x0 (or equivalently ε). Then use the tractable posterior
Training objective (DDPM simplified)
ℒ = 𝔼t,x0,ε[ ‖ ε − εθ(xt, t) ‖2 ] plain MSE between true noise and predicted noise
Why predict ε instead of x0?
  • ε prediction is scale-invariant: target is always unit Gaussian
  • x0 prediction is blurrier at high noise levels
  • Both are equivalent: ε and x0 are related by the closed-form
Session 1 · Reverse 1:05
Time embeddings

The denoiser must know which timestep it's at: at t≈T the input is near-pure noise; at t≈1 it's almost clean. Passing a raw integer works poorly.

emb(t)2i = sin( t / 10000(i / (d/2−1)) )
emb(t)2i+1 = cos( t / 10000(i / (d/2−1)) )
In JAX
half = dim // 2
freqs = exp(−log(10000) · arange(half) / (half−1))
args = t · freqs
return concat([sin(args), cos(args)])
  • Same formula as transformer positional encoding
  • Each timestep gets a unique, smooth fingerprint
  • Then projected: FC(256) → Tanh
  • Added to image features before the MLP trunk

Session 2 · NB04 0:07
The DDPM reverse step
μθ(xt, t) = 1 √αt · ( xt − βt √1 − ᾱt · εθ ) posterior mean: substitute ε̂ for the unknown x0
xt−1 = μθ + √β̃t · z    z ~ 𝒩(0, I)   (z=0 at t=0) β̃t = 1 − ᾱt−1 1 − ᾱt · βt
  • Use εθ to estimate x0, then take one small step toward it
  • Add fresh noise to stay on the learned distribution
  • 1000 steps: slow but accurate
  • The added noise prevents error accumulation
Session 2 · NB04 0:14
DDIM: skip the steps
Key idea
  • DDPM adds noise at each step: Markovian and slow
  • DDIM removes the stochastic term: becomes deterministic
  • Jump from tfrom → tto directly, skipping steps
x̂0 = xfrom − √1−ᾱfrom · ε √ᾱfrom

xto = √ᾱto · x̂0 + √1−ᾱto · ε
In practice
  • Same trained model εθ: no retraining
  • 50 steps gives comparable quality to 1000
  • Deterministic: same seed → same image
Speedup
DDPM: 1000 model calls
DDIM: 50 model calls → 20× faster

Session 2 · NB05 0:40
Classifier-Free Guidance
The idea
  • Add a class embedding to the time embedding
  • During training, randomly replace labels with a null token (p=10%)
  • The model learns both conditional and unconditional denoising
At sampling time
εCFG = εuncond + w · (εcond − εuncond)
Guidance scale w
  • w = 1: pure conditional. Diverse but less sharp
  • w = 3–5: good quality/diversity trade-off
  • w = 10+: very sharp class signal, less variety
Two forward passes per step
One conditional (with label), one unconditional (with null). Blend by w.

Session 2 · NB06 1:08
MLP → Diffusion Transformer
Why the MLP doesn't scale
  • Flattens spatial structure: no locality or equivariance
  • Parameters grow quadratically with resolution
  • No attention: can't model long-range dependencies
DiT (Peebles & Xie 2022)
  • Split image into patches → treat as a token sequence
  • Transformer processes patches with self-attention
  • Conditioning via Adaptive Layer Norm: not concatenation
The data flow
xt [B,H,W,C] ──► PatchEmbed ──► [B,N,d] + pos_embed │ t [B] ──► TimestepEmbedder ──► [B,d] ─┐ ├──► cond [B,d] c [B] ──► class_embed[c] ──► [B,d] ─┘ │ DEPTH × AdaLNBlock(x, cond) │ LayerNorm ──► PatchUnembed ──► [B,H,W,C]
What changes
  • Model: pure-JAX dict → Flax NNX module class
  • Training: jax.value_and_grad → nnx.value_and_grad + nnx.Optimizer
Session 2 · NB06 1:10
Flax: a thin layer over JAX

A high-performance neural network library built directly on top of JAX. JAX's functional paradigm requires manually managing model parameters and state: Flax simplifies building and training models while keeping every JAX transformation (jit, grad, vmap) available.

The problem with plain dicts
  • Params live outside the model: you thread dicts through every function by hand
  • Fine for a 4-layer MLP, painful for a 6-block transformer with attention, norms, and per-block conditioning
flax.nnx.Module
  • Looks object-oriented: layers are attributes, forward pass is __call__
  • Internally it still adheres to JAX's functional requirements
  • Params aren't stored on the object itself: Flax manages them as PyTrees under the hood
  • Model logic stays a pure function, so jit/grad/vmap still work
Session 2 · NB06 1:11
The Flax ecosystem
Built-in layers
Common layers ready to use: Linear, Conv, BatchNorm, LayerNorm, Attention.
Functional RNG
nnx.Rngs handles key-splitting for you: no manual jr.split bookkeeping per layer.
Optax integration
Direct integration with Optax, JAX's gradient-processing and optimization library, via nnx.Optimizer.
Distributed training
Utilities for sharding and replication: the same model code scales from one device to a pod.
Trade-off
More machinery than raw JAX. Worth it once the model outgrows a couple of layers: exactly the DiT you're about to build.
Session 2 · NB06 1:12
Flax vs. PyTorch
PyTorch
Object-oriented and stateful · parameters live on the module and are mutated in place · autograd builds a dynamic graph on the fly · torch.compile is opt-in
Flax (NNX)
Functional under the hood · parameters managed as PyTrees, model logic stays pure · grad transforms the function itself · jit/vmap/pmap are native, idiomatic, and compose

Neither is strictly better. PyTorch's eager, stateful style is easier to debug line-by-line; JAX/Flax's functional style makes transformations like vmap and multi-device scaling fall out for free.
